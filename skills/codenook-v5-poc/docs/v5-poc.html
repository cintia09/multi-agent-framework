CodeNook v5.0 POC

1. 愿景与原则

CodeNook 是对话驱动的本地工作区操作系统。用户在本地目录通过 LLM 对话驱动整个 SDLC 流程。工作区（.codenook/）是状态唯一来源 —— session 无关、设备无关、平台无关。

六大架构原则

  1. 状态在文件，不在会话 —— 所有持久状态落盘
  2. 主会话 = 纯路由器 —— 不干活，只调度
  3. 每个 agent 调用是独立事务 —— 单轮 + 轻量 + 单一职责
  4. 输出传引用，不传内容 —— 文件路径 + summary
  5. 预防胜于清理 —— LLM 不能主动删 context，靠不纳入
  6. HITL 门即压缩点 —— 用户批准后 session 可 /clear + bootstrap

双层架构

(SDLC-specific)

v5.x 聚焦 SDLC 应用层；v6.x 可能引入写作 / 研究 / 运维 profile，共用同一内核层。

三层 state 设计

层路径角色典型大小
1.codenook/state.json工作区级权威索引（bootstrap 必读）~500 字节
2.codenook/tasks/T-xxx/state.json任务级权威状态（仅当前任务读）1–2 KB

3. 主会话 = 纯路由器

稳态 Context 构成

codenook-core.md          (~20K)  始终加载
+ routing table          (在 core.md 内)
+ state.json             (~500 tokens)
+ current_phase_summary  (~200 tokens)
+ user 最近对话           (~500 tokens)
═════════════════════════════════════════
≈ 21–22K tokens（不随阶段数增长）

主会话永远不做

  • ❌ 读大文件（task.md 除外，初次读）
  • ❌ 分析代码 / 生成长文档 / 构造复杂 prompt
  • ❌ 读 knowledge 或 skills 完整内容
  • ❌ 提及 skill 名字（避免平台自动注入污染 context）

主循环伪代码

while True:
    user_input = listen()
    state = read_json(f".codenook/tasks/{current_task}/state.json")
    next_action = route(user_input, state)  # 查路由表
    if next_action.complex:
        dispatch("planner", task_id, phase)  # 派 planner 构造 prompt
        prompt_path = wait_summary().prompt_path
    else:
        prompt_path = write_variable_manifest(task_id, phase)
    result = dispatch(agent_type, f"Execute. See {prompt_path}")
    update_state(result.summary, result.output_path)  # 只收 summary
    if context_usage() > 0.5:
        suggest_user_to_clear()

任务轻量量化

维度限制
单次 agent context≤ 30K tokens
单次 agent 输出≤ 15K tokens
单次 agent 耗时≤ 10 分钟
单次 agent 职责单一
超限拆为串行子任务

4. Prompt-as-File 三层结构

Layer 1: Template（长期固定）

.codenook/prompts-templates/implementer.md  ← 5–10K 完整模板

Layer 2: Variable Manifest（任务特定，~200 tokens）

# .codenook/tasks/T-003/prompts/phase-3-implementer.md

Template: @prompts-templates/implementer.md
Variables:
  task_id: T-003
  phase: implement
  iteration: 1
  dual_mode: serial-A
  upstream_doc: @../outputs/phase-2-designer-summary.md
  user_feedback: @../hitl/phase-2-feedback.md
Output_to: @../outputs/phase-3-implementer.md
Summary_to: @../outputs/phase-3-implementer-summary.md

Layer 3: Agent Self-Bootstrap

  1. 读 variable manifest
  2. 读 template
  3. 读 Variables 里的 @ 引用
  4. 在自己 context 里合成完整指令
  5. 执行
  6. 写 Output_to + Summary_to
  7. 返回 {status, summary, output_path}

Token 代价对比

方式主会话增量 / phase
❌ 内联 prompt5–10K
✅ 主会话写变量清单~200
✅ Planner 构造（不进主会话）~50

5. Validator 模式（自动判定夹层）

主会话连"判断输出好坏"都不做，派给 validator agent，只收布尔结果：

flowchart LR W[Worker Agent] -->|output 文件| V[Validator Agent] V -->|读 output + criteria| V V -->|verdict + reason| M[主会话] M -->|布尔决策| D{pass/fail/needs_human}

Per-phase token 账

环节主会话成本
Worker dispatch~100 tokens
Worker summary 回收~200 tokens
Validator dispatch~100 tokens
Validator verdict 回收~50 tokens
合计~450 tokens / phase

与 HITL 的关系

Phase 完成
  ↓
Validator（自动门，必经）
  ├─ pass → 是否门控 phase？
  │         ├─ 是 → HITL 人工复核
  │         └─ 否 → 直接推进
  ├─ fail → 自动重试 (≤3 次) 失败后 → HITL
  └─ needs_human → HITL

6. 双 Agent 交叉审查

对关键阶段（implement / review）启用 A/B 双 agent 交叉审查，结果由 synthesizer 合成：

Serial 模式

iteration 1: Agent A → output_A → Agent B 审查 → revise
iteration 2: Agent A 修订 → Agent B 再审 → ...
迭代到 B 通过或达上限

Parallel 模式

Agent A & Agent B 并行产出 → synthesizer 合成最终 output

所有迭代产物落 .codenook/tasks/T-xxx/iterations/，由 dispatch-audit check [6] 验证一致性。

7. Workspace 安全（§23）

三大不变量

  1. 所有凭据进 OS keyring —— 工作区文件只存 ${keyring:codenook/<key>} 引用
  2. 每次 session 启动必跑 security-auditor —— 在 greeting 用户之前
  3. verdict=fail 时拒绝任务工作 —— 必须修复后才能继续

跨平台 Keyring

平台backend额外依赖
macOSKeychain无
WindowsCredential Locker无
LinuxSecretService / KWalletlibsecret（通常预装）

Step 2.5（CLAUDE.md mandate）

1. 加载 core.md
2. 写 manifest 派 security-auditor
3. 收 verdict={pass|warn|fail}
4. fail → 拒绝继续；pass/warn → 进入 Step 3 greeting

8. OPT-7 创建时预检（§22）

设计动机：v4.x 一个已知缺陷 —— 双代理模式询问只在任务创建流程里问；上一会话跳过后，dual_mode: null 写入 state.json，本会话推进时编排循环只检查 PHASE_ENTRY_QUESTIONS，不会回溯创建时遗漏的问题。

修复

while True:
    # 创建时必答项预检
    for task in active_tasks:
        if task.dual_mode is None and task.total_iterations <= 1:
            backfill_dual_mode_question(task)
    # ... 主循环

OPT-7 触发条件

total_iterations > 0 AND dual_mode == null AND status NOT IN (done, cancelled)

9. Session 生命周期（§18）

Bootstrap

  1. 用户新 session 或 /clear
  2. CLAUDE.md 加载 → 指向 core.md
  3. core.md 加载 → 主会话成为"CodeNook 操作员"
  4. Step 2.5: 派 security-auditor
  5. 读 history/sessions/latest.md
  6. 读 tasks/*/state.json 知道所有任务状态
  7. 问用户："继续 T-003（phase 3）还是新任务？"

运行

每 phase 结束:
  - 写 outputs/phase-N-summary.md
  - 更新 state.json
  - 触发 distillation
  - 检查 context 使用率

到达 HITL 门:
  - 派 hitl-adapter agent
  - 决策写入 hitl/
  - 主会话建议 /clear（如 >50%）

10. 模型配置（§24）

策略：所有 sub-agent 默认继承主会话模型。任务创建时不询问；只在用户主动「调优 T-xxx 模型」时才进入选择流程。子任务（T-xxx.y）始终继承父任务，永远不被询问。

解析优先级（高 → 低）

1. tasks/<T>/state.json  models[role]      (per-task override)
2. tasks/<T>/state.json  models.default   (per-task default)
3. config.yaml           models[role]      (workspace override)
4. config.yaml           models.default    (workspace default)
5. "inherit"                               (use main session model)

第一个非空值胜出。字符串 "inherit" 在任何层级都意味着「dispatch 时不传 --model，让平台沿用主会话模型」。

Helper 命令

命令作用
resolve <task> <role>打印当前解析结果
list <task>打印所有 role 的解析表
set <task> <role> <value>写入 task 级 override
unset <task> <role>移除 task 级 override

Dispatch 接线

m = resolve_model(T, R)        # 5 级优先级
if m == "inherit":
    Task(role=R, prompt=...)   # 不传 model，平台继承
else:
    Task(role=R, model=m, prompt=...)

Agent profile 本身不读模型；这是 dispatch 时主会话的决策。

调优触发

用户说「调优 T-003 模型」（或类似自然语言），编排器走如下流程：

  1. 读 config.yaml 知道本工作区 pipeline 涉及哪些 role
  2. 派一次 HITL adapter，列出所有 role 与当前解析值
  3. 用户提供 role=value 覆盖（或接受默认）
  4. 编排器写入 tasks/T-003/state.json 的 models
