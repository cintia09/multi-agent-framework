让 AI 助手能接着上次干

一个把任务、记忆、技能塞进项目目录的小框架

给你的 Claude Code 或 Copilot CLI 装一个任务本——
它就住在项目里的一个 .codenook/ 文件夹， 让 AI 干活的过程能存档、能续上、能搬走。

§ 02 · 起因

我为什么折腾这个东西

不是冲着"做框架"去的，就是自己用 AI 助手时一直被几件事硌得难受，想自己写个工具消掉它。

第一件事：写到一半窗口崩了。重开一个会话，AI 又把我半小时前已经否决的方案兴致勃勃地提了一次。我得把"这个不行、为什么不行"再讲一遍。讲完接着写，又过半小时，又崩。

第二件事：上下文越塞越满，到某个点助手会偷偷把前面的对话压缩掉。等我下一次问"我们之前定的接口签名是什么来着"，它会很自信地编一个新的给我。

第三件事：我在公司用 Claude Code，回家用 Copilot CLI，偶尔在服务器上 ssh 进去也想用。三个环境完全不通气，每换一个就等于从零开始——我教过的代码风格、我让它记住的踩坑点，全留在原来那台机器的某个对话窗口里。

盯着这三件事看久了，我发现它们其实是一回事：AI 助手把"会话"当成了工作的天然容器。一切——上下文、记忆、进度——都绑在那个会话上，会话一关就没了。但写代码这种事本来就不是一次会话能搞完的。

所以我想试一下：能不能把工作的最小单位从"会话"换成"任务"？任务有自己的目录、自己的状态、自己的产出物，跟 AI 会话松耦合。会话开关都行，机器换不换无所谓，反正干活的"现场"留在文件里。

§ 03 · 骨架

整体长什么样：三层

先看骨架，后面所有细节都挂在这张图上。一句话概括：你的 AI CLI 在前面传话，一个小 Python 内核在中间走状态机，所有数据落在 .codenook/ 这个目录里。

关键观察：指挥者（也就是你的 AI CLI）不懂业务。它只是看到 tick 返回什么、就照办什么。业务知识全在插件里。这样换 AI、换插件、换机器，三件事互不干扰。
§ 04 · 动起来

跑起来是什么样

把刚才那张静态图动起来。从你跟 AI 说一句话，到任务做完，中间发生了什么。

入口很朴素——你在 AI CLI 里说一句"用 codenook 帮我加个 xxx 功能"。指挥者识别到这个意图，就调一下 codenook task new 创建任务，然后开始一个循环：调 tick，跑一个阶段，看返回值。返回"done"就结束，返回"还要继续"就接着 tick，返回"等你审批"就停下来问你。

每一步都派一个子 agent 去干（用同一个 AI 模型，开个新会话窗口），干完把产出写到一个 markdown 文件里，控制权交还给状态机。下一次 tick 读这个文件、决定下一步。

顺手把三个词解释清楚

上面这张图里出现的术语，其实就三个，后面就不再啰嗦了：phase 是任务里的一步（clarify / design / implement…），每步对应一个 role（也就是给子 agent 的角色提示词，比如"你是 designer，请输出 …"），步与步之间可能有一个 gate——审批节点，状态机走到这里就停，把问题写进 hitl-queue/ 等你回答。你回答完，tick 接着往前走。

"为什么 implement 不停？"——因为写代码这种事最怕被打断切上下文。其他 phase 都是"产物 + 等审"，唯独 implement 是"埋头一口气干完"。

§ 05 · 细节

几个我比较得意的细节

骨架讲完了，挑三个值得多说两句的点。前两个比较好懂，最后一个稍微复杂。

A整个工作空间能搬走

这个最朴素：.codenook/ 里没有任何机器相关的东西，没有数据库、没有后台进程，全是 JSON / YAML / Markdown。所以一行命令就能搬：

# 在 A 机器上
$ tar czf nook.tgz .codenook/
# 拷到 B 机器，解开
$ tar xzf nook.tgz
# 打开任意一个 AI CLI 接着说
$ claude  # 或者 copilot, gemini, ...
> 接着上次那个 T-007 任务往下做

这个用法我自己用得最多——下班拷回家、或者从笔记本扔到台式机。原本得"重新解释一遍"的成本就没了。

B插件就是几个 yaml，告诉内核怎么走

插件其实没什么神秘的。一个插件就是 plugins/<id>/ 下面几个文件：phases.yaml 列出阶段顺序，roles/<name>.md 给每个 role 写角色提示词，hitl-gates.yaml 列出哪些节点要审。内核读这几个文件，照着走。

# plugins/development/phases.yaml （节选）
phases:
  clarify:
    role: clarifier
    produces: outputs/phase-1-clarifier.md
    gate: requirements_signoff
  design:
    role: designer
    produces: outputs/phase-2-designer.md
    gate: design_signoff
  implement:
    role: implementer
    produces: outputs/phase-4-implementer.md
    # 注意没有 gate

同一个内核，挂 development 就能写代码，挂 writing 就能写文章（outline → draft → polish），挂 generic 就是个通用兜底。我没让插件干什么"魔法"——就是把"流程"和"提示词"从代码里挪到了 yaml 和 markdown 里，谁想加新插件抄一份改改就行。

C会自己长记性

这是我最想做但目前做得最粗糙的一块。基本想法是：每个 phase 跑完，状态机会顺手触发三个抽取器，从这个 phase 的产出里挑出"值得记下来的东西"，分别落到 memory/ 的三个子目录：

抽取器自己也是子 agent，跑出来的就是几行 markdown + frontmatter。每条记忆都进 memory/index.yaml，一个轻量目录：

# memory/index.yaml （示意）
- topic: "任务命名走 T-NNN-slug 格式"
  path: knowledge/2025-11/task-id-format.md
  tags: [convention, naming]
- topic: "实现阶段不要插 HITL"
  path: knowledge/2025-11/implement-no-gate.md
  tags: [pipeline, design-rationale]

下次开新任务的时候，指挥者会在开场就把 index.yaml 扫一眼，遇到话题相关的就把对应的 markdown 顺手读进上下文。所以"长记性"在这里不是 RAG，也不是向量检索，就是一个生成型的索引 + 指挥者按需翻阅。粗糙但够用，而且每条都是人能看懂的 markdown，跑歪了能直接手改。

§ 06 · 上手

怎么试一下，以及聊聊

三行命令的事。装好之后，剩下就是在 AI CLI 里说人话。

~/code/my-project 
# 1. 拿源码
$ git clone https://github.com/cintia09/CodeNook.git

# 2. 装到你想用的项目目录里（development 插件够日常用）
$ cd CodeNook && python3 install.py \
    --target ~/code/my-project --plugin development --yes

# 3. 打开 Claude Code / Copilot CLI，进到 ~/code/my-project，
#    然后就跟它说人话：
> use codenook to add a --tag filter to the list command

剩下的事它自己会推动：识别意图 → 创建任务 → 一个 phase 一个 phase 跑下去 → 该问你的时候停下来问。你审完点 ok 就接着走。任务做完，记忆也顺手攒下来了，下次开新任务能用上。

一句话回扣 · 把"会话"换成"任务"，剩下的事自然就顺了。 上下文不会丢、机器可以换、记忆能留下——这三件麻烦事，根都在那个隐含的"会话即工作单位"假设上。
