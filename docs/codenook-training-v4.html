CodeNook 技术文档

装在工作目录里的多智能体任务编排器 (multi-agent task orchestrator)。本文档面向工程师同行，描述 CodeNook v0.25.5 的架构、数据模型、执行流程与扩展接口。

VERSION 0.25.5 · KERNEL python3 · LICENSE MIT
// 01 — Overview

概述

CodeNook 是一个由 CLI 形态的 AI 助手 (Claude Code、Copilot CLI 等) 驱动的、相阶段门 (phase-gated) 的任务流水线编排器。它将助手会话定位为纯指挥者 (conductor)，把状态机、子代理派发 (sub-agent dispatch) 与人工审批闸门 (HITL gate) 下沉到一个 Python 内核 (kernel)，所有运行时状态以 JSON / YAML / Markdown 落盘到 <workspace>/.codenook/。

STATUS · stable STORAGE · plain-text RUNTIME · python3 ≥ 3.9 DEPS · zero (stdlib only)
// 02 — Motivation

设计动机

在 Claude Code、Copilot CLI 等命令行 AI 助手上做工程化协作，会反复撞到三个结构性问题。CodeNook 针对每个问题给出确定性的解法，三者共同构成本项目的存在理由。

问题 (Problem)表现 (Symptom)解法 (Solution)
Session 易失性
session volatility		 会话窗口关闭即丢失上下文；下次启动需重述背景、目标、决议、未决问题。 任务状态外置到 tasks/<T-NNN>/state.json。新会话由内核读取该文件恢复执行点。
上下文受限
context window pressure		 长任务在 token 上限附近触发 /compact，关键决议被压缩或遗忘。 分阶段派发子代理 (sub-agent)，每阶段在独立上下文中执行；产出落盘后由 extractor-batch 抽取到 memory/。
工作空间耦合
workspace coupling		 受限环境 (公司机器、跳板机) 装不上助手 GUI；记忆/插件依附于全局账户，不随仓库迁移。 所有内核、插件、状态、队列、记忆均位于工作目录下 .codenook/，纯文本，可 git / tar 携带。
NOTE CodeNook 不解决模型本身的幻觉、跑题、能力上限等问题。其目标是把 AI 助手的临时会话状态强约束为可审计、可恢复的文件系统状态。
// 03 — Architecture

架构

CodeNook 采用三层架构。各层通过窄接口通信：内核不感知阶段 (phase) 与角色 (role) 的语义；插件不感知 HITL 队列与状态机；工作空间状态对两者均为只读契约。三层共用三个落地物：角色输出文件的 verdict frontmatter、phases.yaml 中的 gate: 字段、以及 state.json。

3.1 内核 Kernel

位于 <ws>/.codenook/codenook-core/，与领域无关 (plugin-agnostic)。导出唯一的 CLI 入口 codenook (经 .codenook/bin/codenook 薄壳转发到 codenook-core/_lib/cli/)。承载状态机引擎 (orchestrator-tick/_tick.py)、HITL 适配器、三个记忆抽取器 (knowledge/skill/config-extractor) 与安装流水线。可通过 python3 install.py --upgrade 原地升级。

3.2 插件 Plugins

位于 <ws>/.codenook/plugins/<id>/，承载领域知识。插件用 phases.yaml 声明阶段目录 (catalogue) 与画像 (profile)，用 hitl-gates.yaml 声明审批闸门，用 roles/<role>.md 声明子代理的引导提示，用 manifest-templates/phase-N-<role>.md 声明每次派发渲染的工单模板。插件目录在安装后只读。

3.3 工作空间状态 Workspace state

位于 <ws>/.codenook/{tasks,memory,hitl-queue}/。所有运行时数据均为纯文本 JSON / YAML / Markdown，可被 git diff 审阅、tar 打包、scp 转移。任务跨会话、跨机器、跨助手客户端可恢复。

// 04 — Concepts

核心概念

本节定义贯穿全文档的术语。每条术语在语义上互不重叠；在代码中对应的实体名、字段名亦严格遵循此处命名。

Plugin 插件
领域工作流的可装载单元。一个插件声明一组阶段 (phases) 与一组画像 (profiles)；同一仓库可并存多个插件。一等公民插件：development (软件工程，11 阶段 7 画像)、writing (长文写作，5 阶段)、generic (低优先级兜底)。
Phase 阶段
流水线中的一个原子步骤。每个阶段绑定一个角色 (role)、一个产出文件 (produces)，可选地绑定一个闸门 (gate) 与若干迭代/扇出/双模标记 (supports_iteration / allows_fanout / dual_mode_compatible)。
implement:
  role: implementer
  produces: outputs/phase-4-implementer.md
  supports_iteration: true
  allows_fanout: true
  dual_mode_compatible: true
  post_validate: validators/post-implement.sh
Role 角色
子代理的引导画像，定义为 plugins/<id>/roles/<role>.md。同一角色可被多个阶段复用 (例如 reviewer 在 review 与 ship 两阶段都担任)。角色文件声明输出契约 (frontmatter_required) 与单行职责。
Sub-agent 子代理
由指挥者按内核派发指令启动的、独立上下文窗口的 LLM 调用。每次派发对应一个阶段；子代理读取角色文件 (system prompt) 与渲染后的工单 (user message)，将其响应写入 tasks/<T>/outputs/phase-N-<role>.md。
HITL Gate 人工审批闸门
Human-in-the-loop。绑定到具体阶段的同步审批点。阶段产出消费完成后，若该阶段配置了 gate:，tick 在 hitl-queue/<task>-<gate>.json 写入待决条目，任务状态置为 waiting，直到 codenook decide 写入 approve | reject | needs_changes 决议。
Envelope 派发信封
tick --json 在派发子代理时返回的 JSON 对象。指挥者据此知道：派给谁、用哪个 system prompt、用哪个 user prompt、产出写到哪个文件、用什么模型。 
{
  "status": "advanced",
  "next_action": "dispatched implementer",
  "dispatched_agent_id": "ag_T-001_implement_1",
  "envelope": {
    "action": "phase_prompt",
    "task_id": "T-001",
    "phase": "implement",
    "role": "implementer",
    "system_prompt_path": ".codenook/plugins/development/roles/implementer.md",
    "prompt_path": ".codenook/tasks/T-001/prompts/phase-4-implementer.md",
    "reply_path": ".codenook/tasks/T-001/outputs/phase-4-implementer.md",
    "model": "claude-sonnet-4.6"
  }
}
Tick 时钟推进
状态机的单步推进操作，由 codenook tick --task <T> 触发。每次调用至多消费一个阶段产出并至多派发一个新阶段；返回 advanced | waiting | done | blocked | error 之一。指挥者循环调用直至 done 或 blocked。
Verdict 裁定
角色输出文件 frontmatter 中必填的字段。常见取值：ok (正常推进)、needs_revision (触发同阶段重派，iteration += 1)、blocked (终止)。tick 据此决定下一步动作。
Conductor 指挥者
用户面对的 LLM 会话本身。它的职责被严格收窄为：识别用户意图、调用 codenook CLI、消费 envelope、派发子代理、中转 HITL prompt 给用户、提交决议。指挥者不解释阶段产出、不挑选插件外的执行路径。
// 05 — Execution Flow

执行流程

任务的全生命周期由 codenook tick 状态机驱动。下图描绘一次 tick 内的判定与转移 (transition)；箭头上的标签为触发条件。

5.1 dispatch → reply → next phase 步骤化

  1. 读取状态。tick 原子读取 tasks/<T>/state.json，校验 schema。若 status ∈ {done, blocked, cancelled, error}，输出摘要并退出。
  2. 消费在飞产出 (若有)。当 state.in_flight_agent 非空，tick 读取 outputs/phase-N-<role>.md，解析 frontmatter 中的 verdict。verdict == needs_revision 且阶段 supports_iteration 时重派同阶段并自增 iteration，达到 max_iterations 时置 blocked。
  3. 运行 post_validate。若阶段声明了 post_validate 脚本，按其退出码决定推进或同上回到 (2) 的迭代分支。
  4. 触发记忆抽取。同步调用 extractor-batch.sh --reason after_phase，将本阶段产出中的可复用条目分流到 memory/{knowledge,skills,configs}/。
  5. 评估闸门。若当前阶段有 gate，写 hitl-queue/<task>-<gate>.json，置 status = waiting，退出。指挥者收到 waiting 后中转 prompt 到用户，调用 codenook decide 写入决议。
  6. 派发下一阶段。从 phases.yaml 加载 role，将 manifest-templates/phase-N-<role>.md 渲染到 tasks/<T>/prompts/，置 state.in_flight_agent，输出 envelope。指挥者据此派发子代理。
  7. 持久化。原子写回 state.json，输出 ≤ 500 字节的 JSON 摘要。

5.2 调用样例

# 1. 创建任务，指定插件与画像
$ codenook task new --plugin development --profile feature \
    --title "add --tag filter to xueba list" --accept-defaults
# → T-009-add-tag-filter-to-xueba-list

# 2. 推进状态机，得到 envelope
$ codenook tick --task T-009 --json
# → { status: advanced, envelope: { ... role: clarifier ... } }

# 3. 指挥者派发子代理写 reply_path 后再次 tick
$ codenook tick --task T-009 --json
# → { status: waiting, hitl: requirements_signoff }

# 4. 解决 HITL 闸门
$ codenook decide --task T-009 --phase clarify \
    --decision approve --comment "scope ok"

# 5. 循环至 done
$ codenook tick --task T-009 --json   # → advanced ...
$ codenook tick --task T-009 --json   # → done
// 06 — Data Model

数据模型

所有运行时状态由四类文件承载。schema 位于 codenook-core/schemas/，cmd_task.set 与 orchestrator-tick 写入前均执行 JSON-Schema 校验。

文件 / 路径格式职责
tasks/<T>/state.json JSON 单个任务的规范状态：当前阶段、画像、迭代计数、状态枚举、子任务链、在飞代理记录、历史日志。
plugins/<id>/phases.yaml YAML 阶段目录 (catalogue) + 画像 (profiles)。tick 据画像解析阶段链。
plugins/<id>/hitl-gates.yaml YAML 闸门定义：trigger 阶段、required_reviewers、人类可读描述。
plugins/<id>/transitions.yaml YAML v0.1 旧格式遗存；v0.2 起由 profiles 派生。新插件可省略。
hitl-queue/<task>-<gate>.json JSON 待决审批条目：prompt、context_files、decision、decided_at、decided_by。
memory/index.yaml YAML 跨任务记忆的人类可读索引。每次抽取后由 knowledge_query 重新生成。
extraction-log.jsonl JSONL 记忆写入审计 (timestamp, task, phase, extractor, target, action)。

6.1 state.json 关键字段

字段 (Field)类型 (Type)语义 (Semantics)
task_idstring形如 T-NNN-<slug>。slug 由 --input / --title 派生。
pluginstring已选插件 id (development / writing / generic / ...)。
phasestring \| null当前阶段 id；首次派发前为 null。
profilestring解析后的画像名 (feature / hotfix / refactor / ...)。
iteration / max_iterationsint / int当前阶段迭代次数及上限。默认上限 3。
dual_modeenumserial (默认) 或 parallel；后者触发并行 N 路派发。
statusenumpending \| in_progress \| waiting \| blocked \| done \| cancelled \| error。
in_flight_agentobject \| null{agent_id, role, dispatched_at, expected_output}，子代理在飞期间非空。
subtasks / depends_on / decomposedarray / array / bool扇出 (fanout) 链路：父任务被 planner 分解后挂起，等待子任务全部 done。
historyarray追加式事件日志。

6.2 state.json 样例

{
  "task_id": "T-009-add-tag-filter",
  "plugin": "development",
  "phase": "implement",
  "profile": "feature",
  "task_type": "feature",
  "iteration": 1,
  "max_iterations": 3,
  "dual_mode": "serial",
  "status": "in_progress",
  "in_flight_agent": {
    "agent_id": "ag_T-009_implement_1",
    "role": "implementer",
    "dispatched_at": "2025-01-12T03:14:22Z",
    "expected_output": "outputs/phase-4-implementer.md"
  },
  "history": [ ... ]
}
CONSTRAINT 指挥者禁止直接编辑 state.json 或 draft-config.yaml。所有变更必须经 CLI (codenook task set / tick / decide)，由内核执行原子写入与 schema 校验。
// 07 — Memory System

记忆系统

每次阶段产出消费完成后，tick 同步触发 extractor-batch 流程。三个抽取器并行运行，按 secret-scan → hash dedupe → similarity check → LLM judge → write/patch → index regen → audit log 流程将可复用条目分流写入 memory/。下次任务启动时，指挥者读取 memory/index.yaml 作为初始上下文。

7.1 三个抽取器

Extractor捕获对象写入路径与 frontmatter
knowledge-extractor 陈述性发现 — 决议、约定、环境注记。 memory/knowledge/<topic>.md · topic, source_task, source_phase, tags, hash, version
skill-extractor 重复出现的命令/脚本模式，足以提炼为可复用技能。 memory/skills/<name>/SKILL.md + 入口脚本
config-extractor 受 schema 约束的配置变更 (默认分支策略、测试运行器等)。 memory/configs/<entry>.md

7.2 调用契约

$ extractor-batch.sh \
    --task-id T-009 \
    --reason after_phase \
    --workspace /path/to/ws \
    --phase implement

# 输出
{
  "enqueued_jobs": ["knowledge", "skill"],
  "skipped": [{"name": "config", "reason": "no schema-bound change"}]
}
非阻塞模式 当 --reason context-pressure 时，extractor-batch 后台化子抽取器并在 ≤ 200 ms 内返回，使指挥者可在 token 水位线触发抽取而不阻塞当前回合。
// 08 — CLI Reference

CLI 参考

所有指挥者操作经由唯一入口 codenook (POSIX 壳 .codenook/bin/codenook，Windows 壳 .codenook\bin\codenook.cmd)。直接调用 codenook-core/skills/builtin/ 下的内部脚本不在受支持契约内。

命令 (Command)说明 (Description)
codenook task new --plugin <id> --title <s> [--profile <p>] [--input <text>]分配新任务 ID 并写入初始 state.json。--input 派生 slug 与首阶段上下文。
codenook task set --task <T> --field <F> --value <V>原子修改 state.json 中的白名单字段，写入前 schema 校验。
codenook task set-profile --task <T> --profile <p>切换画像。仅在首个阶段产出消费前可用。
codenook tick --task <T> [--json]推进状态机一步。--json 模式输出含 envelope 的结构化结果。
codenook decide --task <T> --phase <p\|gate> --decision <approve\|reject\|needs_changes> [--comment <s>]解决 HITL 闸门。--phase 接受阶段 id 或闸门 id。
codenook status [--task <T>]工作空间或单任务状态快照 (含 model / exec mode / phase / status)。
codenook hitl <list\|show\|render\|decide>列出/查看/渲染/决议待审条目。render 产出自包含 HTML 文件。
codenook extract --task <T> --reason <r> [--phase <P>]手动触发 extractor-batch (一般由 tick 自动调用)。
codenook knowledge list [--plugin <id>]枚举 memory/index.yaml 中的条目。
codenook knowledge search <query>按标签/标题/摘要打分排序。
codenook knowledge reindex重建 memory/index.yaml (装机与升级时自动执行)。
codenook chain <link\|show\|detach>父子任务链管理。chain_root 字段由内核自动维护。
codenook preflight校验当前 .codenook/ 安装的完整性 (路径、schema、权限)。

8.1 退出码

  • 0 — 成功。
  • 1 — 运行时失败 (内核异常、文件 IO、子进程非零退出)。
  • 2 — 用法错误或入口问题缺失 (entry-question 未应答)。
  • 3 — 已附加 / 未修改 (幂等命令的 no-op 退出)。
// 09 — Extensibility

扩展机制

新增插件即新增一个目录。安装流水线 (install-orchestrator) 在 G02–G07 共 12 个安全闸门下校验 plugin.yaml schema、id 唯一性、版本约束、签名、依赖与子系统声明，校验通过后将插件目录原子复制到工作空间。

9.2 plugin.yaml schema 摘要

# M2 install-pipeline contract (REQUIRED) ─────────────────────
id: my-domain
version: "0.1.0"
type: domain
entry_points:
  install: phases.yaml
declared_subsystems:
  - domain/my-domain
requires:
  core_version: ">=0.5.0-m5,<1.0.0"

# Routing / packaging surface ─────────────────────────────────
name: my-domain
applies_to: [data-pipeline, etl]
keywords: [airflow, dbt, 数据管道]
examples:
  - "add a backfill job for orders fact"
anti_examples:
  - "refactor the React UI"
supports_dual_mode: false
supports_fanout: true
routing:
  priority: 5
available_skills: [dbt-test, airflow-dryrun]

9.3 phases.yaml 最小样例

phases:
  intake:
    role: intaker
    produces: outputs/phase-1-intaker.md
    gate: intake_signoff

  build:
    role: builder
    produces: outputs/phase-2-builder.md
    supports_iteration: true
    post_validate: validators/post-build.sh

  ship:
    role: shipper
    produces: outputs/phase-3-shipper.md
    gate: ship_signoff

profiles:
  default: { phases: [intake, build, ship] }

// 10 — Appendix

附录：版本与限制

10.1 v0.25.5 当前能力

  • 三个一等公民插件：development v0.2.0 (11 阶段 / 7 画像)、writing v0.1.1、generic v0.1.2。
  • 三种正交并发机制：迭代 (supports_iteration)、扇出 (allows_fanout)、双模 (dual_mode_compatible)。三者可组合。
  • HITL 双通道：终端 prompt + 自包含 HTML 渲染 (codenook hitl render)。
  • 四级模型解析优先级：task > phase > plugin > workspace。
  • 记忆系统：单一持久目的地 memory/ (knowledge / skills / configs)，附 index.yaml 与 extraction-log.jsonl 审计。
  • 双执行模式：sub-agent (默认，独立上下文) 与 inline (指挥者会话内执行，省去派发开销，适用对话密集型阶段)。
  • 跨平台：POSIX shell 与 Windows cmd 双壳，依赖仅 Python 3.9+ 标准库。

10.2 已知限制

  • 不解决 LLM 本身的幻觉、跑题、能力上限。CodeNook 只规约状态形态。
  • 无内置并发锁。多个指挥者并发 tick 同一任务会触发文件竞态；预期单指挥者驱动单任务。
  • router-agent 子系统已被弃用 (since v0.14.0)，将于下一个 major 版本移除。当前流程由指挥者直接驱动 codenook CLI。
  • 双模并行 (parallel_n) 默认上限 3；超出需修改 plugin 的 config-schema.yaml。
  • dual_mode = parallel 与 allows_fanout = true 组合在 planner 阶段会产生 N×M 子任务图，资源消耗需评估。
  • 记忆抽取依赖 LLM 判官 (LLM judge) 步骤，对极小阶段产出可能误判为无可抽取条目。
