§ 02 · The Problem

当前 AI 助手"用着用着就崩"

不是模型不够强，而是工程化的最后一公里没人补。三个真实痛点，几乎每个重度用户都踩过。

💬

痛点 1 · Session 即一切

对话被当作工作流的天然容器。一个稍复杂的任务，跨好几次对话推不动。

"昨天的窗口关掉了，今天打开还得把上下文重新喂一遍。"
📦

痛点 2 · 上下文窗口不可靠

窗口越塞越满，自动压缩裁剪，前面定的方案、踩过的坑、写过的代码风格——说丢就丢。

"明明三十分钟前才确认过的设计，模型现在又来问一遍。"
🔌

痛点 3 · 与环境强耦合

配置和"记忆"散在各家客户端里，换电脑、换工具、上离线服务器，一切清零。

"我的 Cursor 学到的东西，到了 Claude Code 完全用不上。"

现状 vs 期望

§ 03 · The Intuition

把"对话"换成"任务"

隐喻：Session 是一张随手撕掉的便利贴；CodeNook 是公司里那套带工单系统的项目管理流程。

四条核心特性，对症下药

🎯

任务驱动 · 解耦 session

每个任务一个 T-NNN-slug/ 文件夹，state.json 记录所处 phase、verdict、模型选择，关掉窗口、换台电脑都能续上。

🔌

插件式领域技能

development / writing / generic 三个开箱插件，phase + role + gate 三件套定义工作流，AI 不再"瞎猜"该干嘛。

🧠

智能记忆 · 自动沉淀

每个 phase 完成后，三个抽取器并行从 transcript 提炼 knowledge / skill / config，写进 .codenook/memory/。

🧳

工作空间即一切

"Nothing is global"——所有状态在 <workspace>/.codenook/ 下，纯文件、可 diff、可 tar、可换 AI。

一句话总结 · AI 助手不变，但它的"工作单位"变了。
§ 04 · The Architecture

架构总览：三层模型

Conductor 只是协议传话筒，Kernel 是状态机大脑，Workspace 是可携带的"工作记忆体"。三层各司其职，互不渗透。

三层各司其职：用户表达意图 → Conductor 选插件、调 CLI、转达指令 → Kernel 在状态机里前进 → 插件提供领域剧本 → 一切落地为 Workspace 文件。
→ 任何一层换掉，其他三层都不需要重写。
§ 05 · The Runtime

任务的一生：tick 循环 与 11 phase

把静态架构变成动态流程。一行命令背后，是 conductor / kernel / sub-agent 三方的协奏。

development 插件 · feature profile · 11 个 phase

01clarify
02design
03plan
04implement
05build
06review
07submit
08test-plan
09test
10accept
11ship
HITL 的意义 · 人在关键节点把关，机器在重复劳动里加速。 每个非 implement 的 phase 都有一个 gate，verdict 不是 ok 时自动回退迭代。
§ 06 · Memory & Skills

让 AI 越用越懂你

CodeNook 不是又一个工作流编排器——它会自己长记性。每个 phase 完成后，三个抽取器并行从 transcript 里淘金。

三种沉淀 · 各管一段

📚

knowledge/ · 这个项目"是怎么回事"

领域事实、设计取舍、术语定义。下次再做相关任务，conductor 会从 index 命中相关条目。

🛠

skills/ · 我"会做什么"

可复用的操作流程（如"写公众号文章"、"发布到微博"），格式与 Claude Skills / Anthropic SKILL.md 兼容。

⚙️

configs/ · 我"偏好什么"

用户口味（commit 用英文、不要 emoji 标题、push 前先 lint），自动叠加进每个新任务的 conductor 提示。

真实片段 · .codenook/memory/index.yaml

# 自动维护，可读、可 diff、可版本化
version: 1
knowledge:
  - topic: "CodeNook 三层架构"
    summary: "conductor / kernel / workspace"
    tags: [architecture, core]
    path: memory/knowledge/architecture.md
skills:
  - name: "baoyu-post-to-wechat"
    summary: "发布文章到公众号"
    path: memory/skills/baoyu-post-to-wechat/
§ 07 · Portability

工作空间即一切：打包就能搬家

"Nothing is global"——所有状态都在 <workspace>/.codenook/ 下。设计纪律换来的，是几乎为零的迁移摩擦。

对比维度传统 AI SessionCodeNook Task
状态位置 客户端 / 云账号 工作目录文件
迁移成本 导出 + 重新喂上下文 tar + scp
中断恢复 凭对方记忆 + 手动复盘 tick 接着跑
跨工具 几乎不可能 协议相同即可换
合规 / 离线 依赖云端 纯文件 · 可审计 · 可断网

一条命令的搬家流程

$ tar czf my-work.tgz .codenook/
$ scp my-work.tgz remote:~
$ ssh remote
$ tar xzf my-work.tgz
$ codenook status   # T-006 phase=draft 0
$ codenook tick     # 接着跑，零损耗
设计哲学：把"上下文"从 LLM 厂商手里夺回来，交还给文件系统。

痛点 → session 是错误的工作单位，关闭即消失、压缩即丢失、绑定即孤岛。

RECAP · 02

RECAP · 03

收益 → 可恢复、可迁移、可审计。换电脑、换 AI、断网，工作都能继续。
