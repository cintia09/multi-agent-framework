// 01 — Why

AI 编码会话三件麻烦事

单纯把 AI 当聊天框用，会话一关，活儿基本就丢了。

会话易失

关了终端就忘干到哪了；第二天重开要重铺一遍背景。

上下文受限

窗口塞满之后开始幻觉；长任务需要不断手动归纳。

工作空间耦合

换机器、换 AI 客户端、换厂商，状态都得重新建立。

// 02 — Scenario 01

写一篇技术博客

writing 插件 · outline → draft → review → revise → publish · session 中途关掉

  • DAY 1 · 10:00你："用 codenook 帮我写一篇讲单元测试覆盖率的博客。"
  • 10:01助手：挑 writing 插件，问几个问题：受众、字数、风格。回答记入 state.summary。
  • 10:05系统：tick 推进到 outline，派 outliner sub-agent 写大纲。
  • 10:30助手：HITL gate 把大纲 verbatim 给你看。回 approve，推进到 draft。
  • 11:00你：开会前关掉终端。
  • DAY 2 · 09:00你：新开 Claude Code，"continue T-101"。
  • 09:00助手：读 state.json，看到 draft 还在跑，直接接 tick。
  • 11:30系统：review → revise → publish 跑完，沉到 memory/knowledge/。

// 03 — Architecture

三层结构

上面是 AI 助手 (conductor)，中间是 Python 内核 (kernel)，下面是工作目录里的文本文件。

  • Kernel — 插件无关，Python stdlib，提供 CLI、状态机、HITL adapter、记忆抽取器。
  • Plugins — 域知识，development / writing / generic，每个就是一个目录。
  • Workspace — 纯文本运行时状态，可 diff、可 git、可 tar。

// 03 — Concepts

四个关键术语

看懂这四个词，就能读懂任何一份 dispatch envelope。

Plugin

Plugin

一个目录形态的域包，含 phases.yaml / roles/ / manifest-templates/。

Phase

Phase

流水线上的一段，比如 clarify、design、implement，每段绑一个 role。

Role

Role

sub-agent 的人格设定，定义在 plugins/<id>/roles/<role>.md。

Sub-agent

Sub-agent

conductor 派出去执行单个 phase 的子会话，写一份 verdict 标注的产出后退出。

// 03 — Execution

tick 循环

一个任务的执行就是反复调 codenook tick，每次推进至多一个 phase。

  1. conductor 调 tick：codenook tick --json 让状态机推进。
  2. 返回 envelope：含 phase / role / system_prompt_path / reply_path / model。
  3. 派 sub-agent：按 envelope 调起子会话，子会话写 verdict 产出后退出。
  4. 再次 tick：内核读产出、跑 post_validate、抽取记忆，推进或停。

// 03 — HITL Gate

人审闸口

phase 之间挡一道。系统把产出 verbatim 给你看，等你 approve / reject / needs_changes。

当某个 phase 配了 gate，tick 会把任务停在 status: waiting，写一条 entry 进 hitl-queue/。

conductor 把 prompt verbatim 转给你；你回的 approve / reject / needs_changes 通过 codenook decide --task T --phase <id> --decision <verb> 回写。

tick 再继续。整个回路没有任何全局服务，全部存在工作目录里。

// 03 — Memory

记忆系统

每个 phase 跑完后 extractor-batch 自动跑一次，把有复用价值的东西分三类沉淀。

knowledge/

声明式工程事实：决策、约定、环境注记。落到 memory/knowledge/<topic>.md。

skills/

可执行的小技能：重复出现的 CLI / 脚本模式。落到 memory/skills/<name>/SKILL.md。

configs/

配置决定，落到 memory/configs/<entry>.md。三类共用 index.yaml，下次任务先读它。

// 04 — CLI

CLI 速查

日常操作就这几条；conductor 会替你调，知道在干什么就够了。

命令用途
codenook task new --plugin <id> --title …新建任务，分配 T-NNN-<slug>，写初始 state.json。
codenook tick --task T --json状态机推进一步，返回派发 envelope 或 done / blocked / waiting。
codenook decide --task T --phase X --decision V解决一个 HITL gate。V 取 approve / reject / needs_changes。
codenook hitl <list|show|render|decide>查看 / 渲染 / 决议待审 gate；render 输出独立 HTML。
codenook status [--task T]列出 workspace 全部任务或单个任务的当前状态。
codenook extract --task T --reason <r>手动触发记忆抽取（一般 phase 完成后自动跑）。
codenook preflight校验 .codenook/ 安装是否完好。

// 04 — Adopt

如何接入

没有全局守护进程、没有数据库、没有外服。

把 .codenook/ 放进任何一个工作空间，AI 助手就懂这个协议。
STEP 01

克隆仓库

git clone CodeNook

STEP 02

装内核 + 插件

python3 install.py --target ~/ws --plugin development --yes

STEP 03

开 AI 助手

Claude Code 或 Copilot CLI 打开同一目录，bootloader 自动生效。

STEP 04

开口说话

"用 codenook 帮我 …"，conductor 会自己挑插件、起任务、tick。

// 04 — Limits & Roadmap

限制 与 展望

坦白讲清楚边界，比夸大有用。

当前限制

  • conductor 只在支持子会话派发的 CLI 上跑得顺：Claude Code、Copilot CLI 验证过。
  • 插件目前三个：development / writing / generic，覆盖面有限。
  • 记忆抽取靠 sub-agent 自报，质量取决于 role prompt。
  • 并发模型保守：iteration / fanout / dual-mode 各有上限，避免失控。

路线展望

  • 更多一方插件（review / spec / data-pipeline）。
  • 跨任务的记忆检索增强，index.yaml 升级为可查询索引。
  • dual-mode 投票策略可配，支持自定义收敛函数。
  • web HITL UI 收敛到内核，独立 HTML 渲染外可挂浏览器。
