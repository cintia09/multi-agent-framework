把 AI 助手会话钉进文件系统

CodeNook 是一个装在工作目录里的多智能体任务编排器，让 Claude Code、Copilot CLI 这类命令行 AI 助手按工程化的阶段流水线干活，而不是开一次对话掉一次记忆。

VERSION 0.25.5 · KERNEL python3 · LICENSE MIT

本页按"先看场景，再看架构，最后查细节"的顺序组织。如果你只有五分钟，看完第二节的三个场景就够了。

// 02 — Use Scenarios

三个真实场景

场景的关键不是"AI 帮你做了什么"，而是"会话中断之后，活儿还在不在"。 下面三段时间线展示了 session 关掉、换机器、换助手客户端这三种常见的中断方式。

SCENARIO 01

写一篇技术博客

writing 插件 · outline → draft → review → revise → publish · session 中途关掉

  • DAY 1 · 10:00你："用 codenook 帮我写一篇讲单元测试覆盖率的博客。"
  • 10:01助手：挑 writing 插件，问几个问题：受众、字数、风格。回答记入 state.summary。
  • 10:05系统：tick 推进到 outline phase，派 outliner sub-agent 写大纲，落到 outputs/phase-1-outliner.md。
  • 10:30助手：HITL gate outline_signoff，把大纲 verbatim 给你看。你回 approve，tick 推进到 draft。
  • 11:00你：开会前关掉终端。
  • DAY 2 · 09:00你：新开 Claude Code 窗口，"continue T-101"。
  • 09:00助手：读 state.json，看到 phase: draft, status: in_progress，直接接着 tick；不需要重铺背景。
  • 11:30系统：review → revise → publish 全跑完，status: done。整段经历沉淀到 memory/knowledge/。
SCENARIO 02

调研一个开源项目

generic 插件 · 多轮探索 · 公司机器跑一半 → 家里机器接着跑

  • 周三下午你："用 codenook 调研一下 foo 这个仓库，先看架构再列 API 表。"
  • 14:00–17:30系统：跑了 3 个 phase，sub-agent 把要点写到 tasks/T-202/notes/，记忆抽取沉到 memory/knowledge/foo-arch.md。
  • 下班你：把整个项目目录 git push。
  • 晚上你：家里 git pull，开 Copilot CLI。
  • 21:00助手：读 .codenook/state.json 和 memory/index.yaml，列出"workspace 中有 1 个 in_progress 任务：T-202"，直接接 phase 4 继续。
  • 23:00系统：status: done。所有调研产物都还在仓库里，下次想翻随时翻。
SCENARIO 03

重构一个旧模块

development 插件 · feature profile · clarify → design → plan → implement → test · 中途换 AI 客户端

  • 10:00你：在 Claude Code 里说"用 codenook 重构 auth 模块，把 session 改成 JWT"。
  • 10:15助手：clarify phase 跑完，HITL gate requirements_signoff 弹出。你 approve。
  • 11:30系统：design + plan 跑完，进入 implement。implementer sub-agent 改了 7 个文件。
  • 14:00你：Claude 用量到顶。切到 Copilot CLI 同一目录。
  • 14:00助手：读 CLAUDE.md 里的 codenook bootloader 块，认得这个协议。codenook status 列出 T-303 在 implement phase。
  • 14:01你："continue T-303."
  • 15:30系统：test phase 跑完，HITL gate ship_signoff 弹出，你 approve，任务 done。整个流程不依赖任何一家厂商的会话状态。
// 03 — Architecture

三层结构

场景背后的机制只有三层：上面是 AI 助手 (conductor)，中间是一个用 Python 写的内核 (kernel)，下面是工作目录里的文本文件 (workspace state)。三层通过 state.json、role 输出文件、HITL queue 三种 artefact 通讯。

LAYER 1

Kernel

插件无关、版本固定。提供 codenook CLI、状态机、HITL adapter、记忆抽取器、安装管线，全部 Python stdlib，无第三方依赖。

LAYER 2

Plugins

三个一方插件：development（11 phase / 7 profile）、writing、generic。每个插件就是一个目录，含 phases.yaml、roles/、manifest-templates/。

LAYER 3

Workspace

纯文本运行时状态：每个任务一个目录，跨任务的 knowledge / skills / configs 沉到 memory/。可 diff、可 git、可 tar。

// 04 — How it works

tick 循环

一个任务的执行就是反复调 codenook tick。每 tick 一次，状态机推进至多一个 phase：要么派一个 sub-agent 去干活，要么读上一次 sub-agent 的产物决定下一步。

四个关键术语

  • Phase：流水线上的一段，比如 clarify、design、implement。每个 phase 绑一个 role。
  • Role：sub-agent 的人格设定 (system prompt)，定义在 plugins/<id>/roles/<role>.md。
  • Sub-agent：被 conductor 派出去执行单个 phase 的子会话，写一份 verdict 标注的产出文件后退出。
  • HITL gate：phase 之间的人审闸口，把产出 verbatim 给你看，等 approve / reject / needs_changes。

一次完整的派发凭一份 envelope：{ task_id, phase, role, system_prompt_path, prompt_path, reply_path, model }。conductor 按这份 envelope 调起 sub-agent，sub-agent 把答复写到 reply_path，conductor 再回头 tick。整套协议没有任何全局状态，全部存在工作目录里。

// 05 — Details

细节参考

下面五个折叠块分别是 CLI、文件结构、记忆系统、并发模型、扩展指引。需要查的时候打开看。

5.1 · CLI 命令一览
命令用途
codenook task new --plugin <id> --title …新建任务，分配 T-NNN-<slug>，写初始 state.json。
codenook tick --task T --json状态机推进一步，返回派发 envelope 或 done/blocked/waiting。
codenook decide --task T --phase <id> --decision <verb>解决一个 HITL gate。--decision 取 approve / reject / needs_changes。
codenook hitl <list|show|render|decide>查看 / 渲染 / 决议待审 gate；render 输出独立 HTML。
codenook status [--task T]列出 workspace 全部任务或单个任务的当前状态。
codenook extract --task T --reason <r>手动触发记忆抽取（一般 phase 完成后自动跑）。
codenook chain link / show / detach管理父子任务链。
codenook preflight校验 .codenook/ 安装是否完好。

退出码：0 成功 · 1 运行错 · 2 用法错 / 缺 entry-question · 3 无变化。

5.2 · 工作目录结构 
# 安装后 workspace 多出来的就这一个目录
<workspace>/.codenook/
├── codenook-core/          ← 内核 (CLI + builtin skills)
├── plugins/<id>/           ← 已装插件 (read-only)
├── bin/codenook(.cmd)      ← Python shim
├── tasks/<T-NNN>/          ← 每任务一个目录
│     ├── state.json          ← 任务状态 (schema 校验)
│     ├── prompts/            ← 渲染好的 manifest
│     ├── outputs/            ← sub-agent 产物 (verdict)
│     └── audit/dispatch.jsonl
├── memory/                 ← 跨任务沉淀
│     ├── knowledge/<topic>.md
│     ├── skills/<name>/SKILL.md
│     ├── configs/<entry>.md
│     └── index.yaml
├── hitl-queue/             ← 待审 gate
└── extraction-log.jsonl    ← 记忆写入审计

所有文件都是 JSON / YAML / Markdown。可 git 提交、可 tar 打包、可 scp 拷贝；没有数据库，没有后台进程。

5.3 · 记忆系统

每个 phase 跑完后 extractor-batch 自动跑一次，把产出里有复用价值的东西分三类沉淀：

  • knowledge：声明式工程事实（决策、约定、环境注记），落到 memory/knowledge/<topic>.md。
  • skills：可执行的小技能（重复出现的 CLI / 脚本模式），落到 memory/skills/<name>/SKILL.md。
  • configs：配置决定，落到 memory/configs/<entry>.md。

三类共用一份 memory/index.yaml 作为目录索引。下次开任务时 conductor 先读这份索引，知道这个仓库以前做过什么、踩过什么坑。每次写入都进 extraction-log.jsonl，可审计。

5.4 · 并发模型

三个正交机制，可组合：

  • Iteration：supports_iteration: true 的 phase，verdict 为 needs_revision 或 post_validate 失败时，同 phase 重派，iteration += 1，达到 max_iterations（默认 3）转 blocked。
  • Fanout：allows_fanout: true 的 role 在产出里写 decomposed: true 时，tick 拉子任务列表，seed 子 task，父任务 status: waiting 直到子任务全部落地。
  • Dual-mode：dual_mode_compatible: true 的 phase 在 dual_mode: parallel 任务上会派 N 个 sub-agent 并行（默认 N=3），按多数 verdict 收敛。
5.5 · 扩展指引

新增插件就是一个目录：

plugins/<id>/
├── plugin.yaml              ← id / version / 路由优先级
├── phases.yaml              ← phase 目录 + profile 列表
├── hitl-gates.yaml          ← gate id → trigger phase
├── roles/<role>.md          ← role 系统提示 (sub-agent 看)
├── manifest-templates/      ← 每 phase 一份模板
└── README.md / CHANGELOG.md

装入 workspace 用同一个入口：

$ python3 install.py --target ~/code/my-project --plugin <id> --yes

可选附加目录：knowledge/（随插件出货的领域知识）、skills/（可执行小技能，例如 test-runner）、validators/（post_validate 脚本）。详细字段参见 docs/architecture.md §3。
